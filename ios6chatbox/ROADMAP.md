# Roadmap

## 目标

把一个“只能连接 ChatGPT/OpenAI 的旧 iOS 客户端样板”改造成一个：

- 面向 `iOS 6`
- 可配置服务提供商
- 可配置模型
- 以文本聊天为核心
- 结构清晰、适合持续迭代

的原生客户端。

## 总体策略

先做最小改造，不在第一轮重写整套 UI。

优先顺序：

1. 先跑通可配置的聊天链路
2. 再整理持久化和配置结构
3. 最后做增强功能与更强兼容性

## P0: 建立可验证基线

目标：

- 在你的 `Xcode 5.1.1` 环境中确认工程可打开、可编译、可装机
- 在 `iOS 6` 真机上确认基础 UI 可用
- 记录现成 IPA 的行为，作为后续回归基线

交付物：

- 一份真机测试记录
- 一份已知问题清单

风险：

- 现成 IPA 可以运行，但接口已不兼容
- 现成源码可以编译，但运行时网络失败

## P1: 去除 ChatGPT 强绑定

目标：

- 去掉“登录 ChatGPT”产品叙事
- 去掉对固定 OpenAI 账号逻辑的依赖
- 把 app 改成“连接配置”驱动

具体修改：

- 将欢迎页里的登录文案改成“配置 API”
- 将 `/v1/me` 登录校验改成“测试连接”
- 移除所有必须依赖 ChatGPT 账号身份的逻辑
- 将 app 名称、文案和默认标题改成中性名称

验收标准：

- 用户无需“登录账号”
- 用户只需提供配置即可开始聊天

## P2: 抽离配置层

目标：

- 把服务地址、模型名、鉴权信息从散落代码中集中出来

第一版配置项建议：

- `providerDisplayName`
- `baseURL`
- `apiKey`
- `chatPath`
- `chatModel`
- `imagePath`
- `imageModel`
- `extraHeaderName` 可选
- `extraHeaderValue` 可选

默认策略：

- `baseURL` 默认留空
- `chatPath` 默认 `/v1/chat/completions`
- `imagePath` 默认 `/v1/images/generations`
- 不预设只属于单一厂商的文案

验收标准：

- 发送请求时不再引用硬编码 `https://api.openai.com`
- 所有请求参数都来自集中配置

## P3: 改造网络层

目标：

- 将网络请求从“固定 OpenAI 请求构造”升级为“通用 OpenAI-compatible 请求构造”

第一版范围：

- 文本聊天
- Bearer Token 鉴权
- JSON 请求与响应解析
- 统一错误提示

实施建议：

- 新建一个配置读取模块
- 新建一个 endpoint 构建模块
- 保留当前老系统可用的请求方式
- 先不大改 UI 层，只替换底层请求构造和响应解析

第一版暂不做：

- 流式输出
- 多协议 provider adapter
- OAuth
- 复杂重试机制

验收标准：

- 可连接自定义 `baseURL`
- 可正常完成一轮文本问答

## P4: 调整持久化

目标：

- 把会话存储从临时目录迁移到长期存储位置
- 为后续配置、多 provider、历史管理留下空间

建议：

- 会话文件迁移到 `Documents` 或 `Library/Application Support`
- 配置项仍保留在 `NSUserDefaults`
- 头像缓存与会话文件分离

验收标准：

- 重启 app 后历史仍稳定存在
- 系统清理临时目录不会导致聊天记录丢失

## P5: 产品化重命名与文案收口

目标：

- 让产品身份从“ChatGPT 客户端”变成“老设备 AI chat 客户端”

修改范围：

- 应用名
- 启动文案
- 设置页标题
- 默认会话标题
- 错误提示中的品牌词

验收标准：

- 用户不会误以为这是官方 ChatGPT 客户端

## P6: 第一批增强功能

建议优先级如下：

1. 配置多个 provider
2. 手动测试连接按钮
3. system prompt
4. temperature / max tokens
5. 更好的错误展示
6. 会话重命名与搜索

## P7: 第二批增强功能

等 MVP 稳定后可以再做：

- 图片理解
- 图片生成独立入口
- 自定义 headers
- 代理与中转支持
- Markdown 渲染改进
- 日志导出
- 调试面板

## 非目标

当前阶段不建议投入的方向：

- 完整重写 UI
- 过度抽象的 provider 插件系统
- 云同步
- 用户账号体系
- 复杂本地数据库

## 第一周建议节奏

1. 建立代码基线并确认能本地编译
2. 改欢迎页文案和登录流程
3. 接入 `baseURL + apiKey + chatModel`
4. 打通一轮自定义 provider 文本聊天
5. 修复真机上的错误提示和持久化问题
